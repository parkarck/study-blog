하네스(Harness) 엔지니어링 설계 이론의 활용과 실험을 통한 효율적 개발 방법론 탐구

## 2. 주요 개념 정리
- **하네스 엔지니어링**: 다양한 에이전트가 협업하여 제품 개발을 최적화하는 방법론
- **주요 에이전트**: 플래너, 제너레이터, 이밸루에이터
  - **플래너**: 사용자 프롬프트를 제품 스펙으로 확장
  - **제너레이터**: 스펙을 구현 가능한 코드로 변환
  - **이밸루에이터**: 사용자 입장에서 테스트 및 피드백 제공

## 3. 세부 설명
- **플래너의 역할**: 제품의 하이레벨 설계에 집중, 세부 기술 구현 방지
- **제너레이터의 역할**: 스펙에 따라 기능을 구현, 스프린트 방식으로 작업
- **이밸루에이터의 역할**: UI, API, DB 상태 테스트 및 피드백 제공
- **스프린트 구조**: 제너레이터와 이밸루에이터가 협상하여 목표 설정 후 작업

## 4. 예시 / 적용
- **2D 레트로 게임 메이커 개발**: 
  - 단일 에이전트 vs 풀 하네스: 후자에서 더 높은 품질과 기능 확보
  - 하네스 적용: 16개 기능으로 확장, 실제 게임 플레이 가능

- **DAW 음악 제작 프로그램 개발**:
  - 스프린트 구조 제거 후 일관된 코드 작성
  - 이밸루에이터의 피드백으로 디테일 보완

## 5. 시사점

## Timeline

**Core Process:**

1. **플래너 에이전트 사용 및 역할 정의**  
   - 플래너는 사용자가 제공한 짧은 프롬프트를 전체 제품 스펙으로 확장합니다. 기술적 세부사항보다는 제품 컨텍스트와 하이레벨 설계에 집중하며, AI 기능을 자연스럽게 통합하는 방법을 찾습니다.

2. **제너레이터 에이전트의 기능 구현**  
   - 제너레이터는 플래너가 제공한 스펙을 기반으로 하나씩 기능을 구현합니다. 리액트, 바이트, 패스트 API, SQL라이트를 사용하며, 기스로 버전 관리를 합니다.

3. **이밸루에이터 에이전트의 테스트 및 피드백 제공**  
   - 이밸루에이터는 실제 사용자가 앱을 사용하는 것처럼 테스트하여 UI, API, DB 상태를 검증하고, 버그를 찾아내어 상세한 피드백을 제공합니다.

4. **스프린트 계약 협상 및 실행**  
   - 스프린트 시작 전 제너레이터와 이밸루에이터가 협상하여 완료 조건과 검증 방법을 합의합니다. 이 후에 코딩을 시작합니다.

5. **실험 및 개선 사례 분석**  
   - 두 가지 사례를 비교 분석하여, 하네스 버전이 더 복잡하고 비용이 높지만 결과물이 뛰어나다는 것을 보여줍니다. 이후 모델 업데이트에 따른 하네스의 재점검 필요성을 강조합니다.

6. **업데이트된 모델에 대한 하네스 구조 최적화**  
   - 오프스 4.6 업데이트에 따라 스프린트 구조를 제거하고, 제너레이터는 연속적인 코딩, 이밸루에이터는 최종 검수로 구조를 단순화합니다.

7. **최종 결과물 및 인사이트 공유**  
   - 브라우저 기반 DAW 제작 프로그램의 결과를 공유하며, 모델의 발전이 하네스의 필요성을 줄이는 것이 아니라 조합의 변화를 가져온다는 인사이트를 강조합니다.
